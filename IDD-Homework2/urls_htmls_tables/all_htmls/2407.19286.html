Abstract

Federated learning is a distributed learning setting where the main aim is to train machine learning models without having to share raw data but only what is required for learning. To guarantee training data privacy and high-utility models, differential privacy and secure aggregation techniques are often combined with federated learning. However, with fine-grained protection granularities the currently existing techniques require the parties to communicate for each local optimisation step, if they want to fully benefit from the secure aggregation in terms of the resulting formal privacy guarantees. In this paper, we show how a simple new analysis allows the parties to perform multiple local optimisation steps while still benefiting from joint noise scaling when using secure aggregation. We show that our analysis enables higher utility models with guaranteed privacy protection under limited number of communication rounds.

1 Introduction

Federated learning (FL) [31, 25] is a common distributed learning setting, where a central server and several clients holding their own local data sets collaborate to learn a single global model. The main feature in FL is that the clients do not directly communicate data, but only what is required for learning, e.g., gradients or updated model parameters (pseudo-gradients).

While FL satisfies the data minimization principle, i.e., only what is actually needed is communicated while the actual raw data never leaves the client, it does not protect against privacy attacks such as membership inference [38] or reconstruction [19, 51]. Instead, training data privacy is commonly ensured by combining differential privacy (DP) [17]), a formal privacy definition, and secure multiparty computation (MPC), [50] with FL (see, e.g., [25]).

DP is essentially a robustness guarantee for stochastic algorithms, which guarantees that small perturbations to the inputs have small effects on the algorithmsâ€™ output probabilities. What constitutes a small perturbation depends on the chosen protection granularity: the same basic DP definition can be used for ensuring privacy on anything from single sample to entire data set level. In turn, MPC protocols can be used to limit the amount of information an adversary has about computations. In FL, secure aggregation (SecAgg) protocols, a specialised form of secure computation that requires significantly less resources than general MPC, are commonly used for communicating model updates from the clients to the server, which can result in provably better joint DP guarantees than is possible to achieve by any single client in isolation.

Under the general FL setup, two main alternatives are commonly considered: cross-device FL and cross-silo FL. In cross-device FL, each client is assumed to have a small local data set, while the total number of clients is large, e.g., thousands or millions. In the cross-silo case, the total number of clients is small, for example, a dozen, but each client is assumed to have a larger local data set. In this paper, our running example is standard cross-silo differentially private FL (DPFL) where the clients communicate all updates to the server using SecAgg.111Instead of considering any specific SecAgg implementation, in this work we mostly assume an idealised trusted aggregator. In this setting, the most useful DP protection granularity is typically something strictly more fine-grained than client-level: when clients are, e.g., different hospitals or banks, there are typically several individuals in a single clientsâ€™ local data set and the protection granularity needs to match the use case.

While client-level granularity in DPFL is, at least in principle, straightforward to combine with SecAgg, more fine-grained granularities present a problem: using existing techniques one has to choose between i) having joint DP guarantees with less noise due to SecAgg but with all clients using only a single local optimisation step per FL round, and ii) having more noisy local DP (LDP) guarantees that do not formally benefit from SecAgg while allowing the clients to do more local optimisation steps per FL round. Both of these options have significant drawbacks: the amount of server-client communications is typically one of the first bottlenecks that limit model training in FL, while LDP guarantees regularly require noise levels that heavily affect the resulting model utility. In this paper we show that this trade-off is not unavoidable but can be largely remedied by a simple new analysis of the problem.

2 Related Work

There is a significant amount of existing work focusing on the general problem of combining DP with FL, although the focus has mostly been on the cross-device FL setting with user-level DP. To the best of our knowledge, while the combination of DPFL with SecAgg is certainly not novel (see, e.g., [45, 25, 21, 42, 49]), there is no existing work on the privacy analysis when the clients do multiple local optimisation steps fine-grained sample-level DP granularity and communicate the results via SecAgg.222Note that [45, AlgorithmÂ 4] seem to state a weaker, specialised version of our results, i.e., they use several local optimisation steps with sample-level DP and SecAgg in FL, while scaling the noise jointly over the clients. However, as noted in [30], the approach in [45] would require a separate proof of privacy beyond what is actually provided in the paper.

Considering the existing work in more detail, we can distinguish several main lines of research. There are many papers proposing novel learning methods for FL, assuming sample-level DP and joint noise scaling with SecAgg. While the existing work only uses a single local optimisation step (see, e.g., [21, 30, 42, 49]), our analysis can be utilised in this setting to enable multiple local steps generally for many such methods.

Another clear line of work has focused on introducing novel discrete DP mechanisms that can be used with additively homomorphic encryption techniques, which typically operate on the group of integers with modulo additions. [2] proposed a binomial mechanism that provides DP using discrete binomial noise. Improving on the binomial mechanism, [11] proposed a discrete Gaussian mechanism, while [1] introduced a Skellam mechanism and [14] a Poisson-binomial mechanism, both of which improve on the discrete Gaussian, e.g., by being infinitely divisible distributions: the sum of Skellam/Poisson-binomial distributed random variables is another Skellam/Poisson-binomial random variable. Our work is not focused on introducing new DP mechanisms, but our analysis allows for using many different DP noise mechanism. In particular, our analysis allows for joint noise scaling under SecAgg including when using infinitely divisible DP mechanisms, such as the Skellam mechanism, with pseudo-gradients and fine-grained DP protection level.

One theme that has seen much recent activity is the combination of DPFL with quantization techniques to minimize the amount of bits communicated by the clients. [2] combined quantisation with the binomial mechanism, while [24] used discrete Gaussian with randomised rounding for DPFL, and [1] did the same with the Skellam mechanism. Again, one main motivation stated for introducing the Poisson-binomial mechanism [14] is that it requires the clients to communicate less bits as the privacy gets tighter, contrary to what happens, e.g., with the discrete Gaussian mechanism. Furthermore, in the tight communication budget regime, [23] proposed stochastic DP sign-SGD, which provides LDP guarantees and communicates 1 bit per client gradient dimension. In the same vein, [12] proposed the Minimum Variance Unbiased (MVU) mechanism for combining DPFL and heavy compression, while [20] introduced Interpolated MVU (I-MVU), that improves on MVU for DPFL especially in the case of high-dimensional data by relaxing the requirement of unbiasedness. While in principle it is possible to combine our analysis with quantisation techniques which are compatible with SecAgg, since the model updates in our case are pseudo-gradients and not gradients, this is not entirely straightforward.

Another alternative approach to improve communication-efficiency of DPFL is to compress the updates directly instead of or in addition to quantising them. On this front, [44] introduced an approach based on relative entropy coding, where the DP guarantees are derived based on the inherent randomness of the compression method without needing any additional DP noise mechanism. However, their approach does not support using SecAgg for joint noise scaling but results in LDP guarantees, and so is not directly compatible with our approach. Finally, [13] proposed and analysed a linear compression scheme, that is compatible with SecAgg, based on sparse random projections. Due to linearity, our approach should be compatible with their method, and lead to reduced communications while still allowing for joint noise scaling.

We leave the detailed consideration and comparison of the possible methods for reducing the required communication budget beyond what is possible by pushing more optimisation steps to the clients while reducing FL rounds for future work.

In summary, while many of the contributions cited above, e.g., novel DP mechanisms, are not limited to cross-device FL, all the experiments and use cases mentioned in the cited papers that are compatible with SecAgg and use multiple local steps only consider joint noise scaling with user- or client level DP in cross-device FL. In contrast, we focus on more fine-grained DP granularities, e.g., on sample-level DP. As we discuss in SectionÂ 3, combining sample-level DP with multiple local steps and joint noise scaling using SecAgg with good utility requires a novel privacy analysis. The main aim of this paper is to provide such an analysis.

3.1 Trust Model

In this paper, we assume an honest-but-curious (hbc) server and that all the clients are fully honest. The latter assumption can be easily generalised to allow for hbc clients with some weakening to the relevant privacy bounds: with Nğ‘N (non-colluding) hbc clients, since any client could potentially remove its own noise from the aggregated results, the noise from the other Nâˆ’1ğ‘1N-1 clients needs to guarantee the target DP level. In effect, to allow for all hbc clients, we would need to scale up the noise level somewhat (see, e.g., [22] for a discussion on noise scaling and for formal proofs).

In principle, the same technique can also protect against privacy threats in the case of including some fully malicious clients in the protocol (i.e, simply scale the noise so that the hbc clients are enough to guarantee the required DP level). However, in this case the required level of extra noise will increase quickly with the number of malicious clients leading to heavier utility loss. With malicious clients, there would also be no guarantee that the learning algorithm terminates properly.

5 Experiments

In this section we empirically demonstrate the benefits of being able to run more than a single local optimisation step under limited communication budgets. Our chosen settings mimic a typical cross-silo FL setup: there are a limited number of clients, each having a smallish local database. The clients have enough local compute to run optimisation on the chosen model, while the server-client communications required for updating the global model are the main bottle-neck.

In this setting, it makes sense to try and push more optimisation steps to the clients while reducing the number of global updates (FL rounds). We also assume that the clients send their local updates via some trusted aggregator (which we only assume and do not implement in practice in the experiments). For more details on all the experiments, see AppendixÂ Appendix.

We first train a small convolutional neural network (CNN) on Fashion MNIST data [47], that is distributed iid among 10 clients. We use the CNN architecture introduced by [35] (see also [43]). Comparing the results for a single local step as opposed to 111 local epoch, it is evident that being able to take more local optimisation steps (as allowed by TheoremÂ 4.7) can bring considerable utility benefits under a fixed privacy and communication budget.

Overall, assuming fixed communication and privacy budgets, we would expect the benefits from being able to run more local steps to be more accentuated with more complex models, while for simple-enough models even a few local steps might give good results. 6 Discussion

In this paper we have shown how to combine multiple local steps in DPFL using fine-grained protection granularities with SecAgg, and demonstrated that this can bring considerable utility benefits under communication-constrained settings.

While one of the main points in using FL is that limitations resulting from a bottle-neck on the maximum number of communications or from limitations on the available bandwidth between the parties can be relaxed by pushing computations to the clients, it is still an important problem to try and reduce the required communication budget further; in the present work, all model updates require 323232 bits per parameter to communicate. Taking the number of FL rounds as given, to reduce communications requirements the clients need to communicate less bits per FL round.555Typically, clientsâ€™ upload bandwidths are more constrained compared to available download bandwidths.

As mentioned in SectionÂ 2, possible solutions that have so far been considered in the cross-device DPFL setting include quantisation as well as dimensionality reduction (random projections). We leave the exploration of these methods in the cross-silo setting as open questions.

In this paper, we do privacy accounting based on RDP [33]: we account for the privacy of each individual local optimisation step with joint noise from all the clients selected for a given FL round. When the clients use Poisson subsampling to sample minibatches (we assume each client uses the same probability for including any individual sample in the minibatch), we use standard RDP privacy amplification results. In practice, we use the RDP accountant implemented in Opacus [52], as well as bounds for Skellam mechanism from [1] and tight RDP amplification by Poisson subsampling [41]. We calculate the privacy cost of the entire training run in RDP, and then convert into ADP using [33, Proposition 3]. Note that, as is common in DP research, we do not include the privacy cost of hyperparameter tuning in the reported privacy budgets (see e.g. [43] for some reasoning on this practice).
