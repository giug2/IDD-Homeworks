Despite being a very popular approach in a federated learning setting, DecAvg is expected to suffer, in a decentralized environment, from the lack of central coordination. In standard FedAvg, the server sends a common initial model to all nodes, which then starts the learning with common parameters. Without the common initialization, local models are expected to associate different features with different neurons (due to the permutation invariance of the hidden layers of neural networks). When this happens, coordinate-wise averaging can be detrimental (because nodes are averaging weights that do not match the corresponding learned features). However, the lack of a common initialization can only be solved by either forcing the nodes to coordinate in a decentralized way before the learning phase or exploiting strategies that do not require coordination. We believe that an initial coordination round is not suitable for dynamic decentralized scenarios (percolating the common initialization across the network might be time-consuming, the nodes of the networks may come and go, etc.). For the second approach, we discuss below some possible solutions.

4 Baselines and benchmarks

We compare decentralized learning against the following baselines.

Central learning

All the data are on a central server and the reference model is trained on it. There is a single model in the network, which we denote with F​(⋅;𝐰g)𝐹⋅subscript𝐰𝑔F(\cdot;\mathbf{w}_{g}).

Federated learning - FedAvg

This is a federated learning scenario where data are scattered across client devices, which communicate with a central coordinator (server) to accomplish the learning task according to the FedAvg strategy (in short, at each communication round, the server sends the current global model to the clients, which in turn train the model on their local data, then send the model back to the server, which updates the global model with the average of the received models). We denote the global model collected by the federated learning benchmark with F​(⋅;𝐰f)𝐹⋅subscript𝐰𝑓F(\cdot;\mathbf{w}_{f}).

Local learning

There is no central server. Client devices own a fraction of the dataset111No overlapping is allowed. (it could be IID or non-IID) and train their local models on this data.

DecAvg with initial coordination

We assume that nodes are able to coordinate initially to start learning from a common model. This is not generally feasible in realistic settings (unless the network is very small and stable) but it still provides a good benchmark for what decentralized learning can achieve under ideal conditions.

DecTS-Oracle

The upper bound to a teacher-based learning is when the teacher is as good as they get, i.e., the teacher model is exactly the centralized model obtained with Central learning. This baseline is not meaningful in practice, but it provides an estimate of the potential of TS learning: if DecTS-Oracle does not beat another target strategy, there is no designing an alternative teacher that can be that [@Lorenzo: actually, this might not be true? Some noise could prove useful?].

CFA-GE

Consensus-based Federated Averaging with Gradient Exchange by savazzi_federated_2020. It is an improved version of CFA where devices exchange the models and the gradients. These two pieces of information are used to improve the algorithm’s convergence. This version shows superior performance w.r.t. CFA, although at a greater computational and communication cost since the devices have to compute the gradients of each model received from the neighbourhood and communicate them back to the corresponding device asynchronously.

Whatever the specific decentralized learning strategy, the goal of decentralized learning is to approach the performance of its centralized counterpart. More realistically, given that the latter is an ambitious goal for federated learning itself, we strive to perform similarly to federated learning.
